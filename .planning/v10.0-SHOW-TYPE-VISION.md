# v10.0 ‚Äî Show Type / Episode Continuity ‚Äî Vision Notes

**Status:** Vision captured (not yet planned). Pre-work for `/gsd:new-milestone`.
**Captured:** 2026-06-17 (conversational discovery with the user)
**Supersedes prior thinking:** none ‚Äî new feature area.

---

## The problem

Today a `Show` is a collection of episodes (`Project` rows with `show_id` + `episode_number`)
that share a bible (`bible_characters`, `bible_world_setting`, `bible_season_arc`,
`bible_tone_style`) and an `episode_duration_minutes`. But when the AI writes/generates an
episode, it does so **blind to the other episodes** ‚Äî there is no notion of whether episodes
are connected (episode 3 should know what happened in 1-2) or independent (anthology).

The user wants to declare, per show, **how episodes relate**, and have that change how the AI
writes, the wizard flow, and review.

---

## Core decision: ONE axis ‚Äî `continuity_mode`

After weighing single-enum presets vs. multi-axis vs. preset+override, we deliberately chose
**a single behavioral axis** because the only thing that actually changes engine behavior is
**what prior context the AI receives**. "Scale" (microseries vs full series) is metadata
(`episode_duration_minutes` already exists) and does NOT earn a place in the type.

`Show.continuity_mode` ‚àà:

| Mode | What the AI sees when writing episode N | Use case |
|------|------------------------------------------|----------|
| `connected` | season arc (bible) + **auto-generated summaries of prior episodes** (ordered by `episode_number`) + character state | microseries, connected series |
| `anthology` | only the shared bible (world/tone) ‚Äî NOT other episodes' plots | independent episodes |
| `standalone` | single episode, no season | feature film |

**Presets are a UI affordance, not a data axis.** Labels like "Microserie / Serie conectada /
Antolog√≠a" can sit on top as visual shortcuts that set `continuity_mode` (+ a sensible default
duration), but the model stores only the mode.

---

## What it touches (the 4 layers the user cares about)

1. **Data model**
   - `shows.continuity_mode` (enum: connected | anthology | standalone; default decided at planning).
   - `projects.episode_summary` (TEXT, AI-generated, nullable).
   - `projects.episode_summary_stale` (Boolean, default False) ‚Äî mirrors the existing
     `breakdown_stale` / `shotlist_stale` pattern.
   - Migration via `delta/NNN_*.sql` (idempotent, applied on boot ‚Äî see Phase 62 mechanism).

2. **Generation (the high-value part)**
   - In `connected` mode, the episode-writing prompt is enriched with: the season arc +
     the `episode_summary` of all prior episodes (ordered by `episode_number`, the reliable key ‚Äî
     NOT positional; see [[screenplaycontent-no-reliable-order]]).
   - In `anthology`, inject only the bible. In `standalone`, no cross-episode context.
   - Touch points: `openai_service.py` / `anthropic_service` prompt builders, wherever
     `bible_context` is currently assembled (`breakdown_service.py:312` shows the existing
     `bible_context` injection pattern to mirror).

3. **Wizard / creation flow**
   - When creating/editing a show, pick the continuity mode (presets as shortcuts).
   - Connected shows surface the season-arc step earlier; anthology hides cross-episode steps.

4. **Auto-summary + invalidation (the subtle part)**
   - When an episode is "completed", the AI generates its `episode_summary` and stores it.
   - **CRITICAL ‚Äî invalidation:** editing an episode after its summary exists must set
     `episode_summary_stale = True` so the summary is regenerated before it feeds later episodes.
     Without this, episode 3 would be written against a stale summary of an edited episode 2 ‚Äî
     the same class of bug that bit twice with ScreenplayContent ordering. The staleness flag +
     regenerate-on-next-use is the structural mechanism that prevents it.

---

## Decisions locked (so downstream planning doesn't re-ask)

- **D1:** Single axis `continuity_mode`, not multi-axis, not closed presets. (User: "conf√≠o en vos" ‚Üí recommended & accepted.)
- **D2:** Prior-episode context is **AI-auto-summarized**, not hand-written, not full-script. (User chose "la IA resume sola.") Full prior scripts rejected (token blow-up on long seasons); manual summaries rejected as the default (user wants it to feel magic).
- **D3:** Summaries must be invalidated on episode edit (`episode_summary_stale`), regenerated lazily.
- **D4:** "Scale" (micro vs standard) stays metadata (`episode_duration_minutes`), NOT part of the type.
- **D5:** Presets are pure UI sugar over the single mode.

## Explicitly deferred (NOT this milestone)

- Automatic continuity-inconsistency detection ("character X is dead in ep2 but appears in ep4").
- Multi-season support (seasons as a layer between show and episode).
- Hand-editable summaries / summary review UI (auto is the default; manual editing is a later nicety).
- "Scale" as a behavioral concept beyond the existing duration field.

## Open questions for the planning phase

- Default `continuity_mode` for existing shows on migration (likely `connected` to preserve
  the implicit "season" intent, or `anthology` as the safest no-behavior-change default ‚Äî decide at planning).
- Where exactly the auto-summary fires (on explicit "complete episode" action vs. on first
  request for a later episode's context) ‚Äî affects cost and UX.
- Token budget / truncation policy when a connected season has many episodes (summaries are
  bounded, but still need a cap).

---

## Why this is the right size

The user marked all four layers (generation, structure, wizard, review), but the leverage is
concentrated in **generation + auto-summary**. Starting with one clean axis (continuity) reuses
what already exists (`bible_season_arc`, `episode_number` ordering, the `*_stale` pattern) and
avoids painting into a corner. Presets and scale can be layered on later without a model change.
