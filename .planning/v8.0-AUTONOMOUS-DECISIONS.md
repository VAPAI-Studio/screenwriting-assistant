### Phase 55 — MCP Foundation

**D-55-A — Library pin: official `mcp` SDK (`mcp>=1.27.2,<2.0`), NOT standalone `fastmcp`.**
- **Why:** Verified the actual dependency floors of `mcp==1.27.2` against the installed environment. It needs `starlette>=0.27` (have 0.36.3 ✓), `pydantic>=2.11` (have 2.13.4 ✓), `uvicorn>=0.31.1` (have 0.27.1 ✗ — small bump), `httpx>=0.27.1`. So the official SDK works with the EXISTING FastAPI 0.110 / Starlette 0.36.3 — **no FastAPI/Starlette bump required**, which is the lowest-risk path. `fastmcp` (standalone) tends to demand newer Starlette and adds a second dependency tree; not worth it for our needs. The official SDK bundles FastMCP v1 at `mcp.server.fastmcp` and exposes `streamable_http_app()` + a `TokenVerifier` hook for static-bearer auth.
- **Reversible?** Yes — swap is localized to the `mcp/` module + requirements pin if `fastmcp` ergonomics turn out to be needed.

**D-55-B — Only dependency bump: `uvicorn` 0.27.1 → `>=0.31.1`.** Everything else `mcp` needs is already satisfied. **Why:** minimum change to admit the MCP SDK. **Reversible?** Yes (pin). **Risk:** low — uvicorn minor bump; will smoke-test app startup after.

**D-55-B2 — Dependency conflict resolved by pinning starlette/sse-starlette, NOT by bumping FastAPI. ⚠ REVIEW THIS ONE.**
- **What happened:** installing `mcp>=1.27.2` let pip pull the newest `sse-starlette` (3.4.4) which dragged in `starlette 1.3.1`, conflicting with FastAPI 0.110.0's `starlette<0.37` cap. The research assumed "no FastAPI bump needed" because `mcp` itself only needs `starlette>=0.27` — true, but the transitive `sse-starlette` latest broke it.
- **Decision:** pin `starlette>=0.36.3,<0.37` and `sse-starlette<2.2` in requirements.txt so the resolver keeps the EXISTING starlette 0.36.3. Verified: `mcp 1.27.2` + `starlette 0.36.3` + `sse-starlette 2.1.3` install clean, `from mcp.server.fastmcp import FastMCP` imports, and `FastMCP(...).streamable_http_app()` returns a Starlette app. **No FastAPI bump.**
- **Why this over bumping FastAPI:** FastAPI 0.110→0.115+ is a large jump for an app with a custom middleware stack and many routes; risky to do unattended. Pinning starlette is surgical and reversible.
- **Reversible?** Yes — if a future MCP feature needs newer starlette, the alternative is the FastAPI bump (its own mini-phase with a full regression run). Flagged for you to decide if you'd rather modernize FastAPI now.
- **Open follow-up:** the install was done live in the running container; requirements.txt is updated, but a clean `docker compose build` should be run to confirm the pinned set resolves from scratch (will verify in execution).

**D-55-C — Static-bearer client spike (GO/NO-GO) verified at API level — PASSED.** Verified via an in-process integration test (`test_mcp_foundation.py`) using the official `mcp` SDK's `streamable_http_client` over an httpx ASGI transport against the mounted `/mcp`: `initialize` + `tools/list` + a `whoami` tool call round-trip with a real `sa_<key>` static bearer; whoami returns the key's owner (MCPF-04); `request_count` increments via the MCP path (MCPF-03); a missing bearer is rejected (MCPF-02). **So the static-bearer transport+auth foundation WORKS.** GUI-client UAT (Claude Code `claude mcp add --transport http`, Claude Desktop, Hermes) remains PENDING USER — documented in CLIENT-SETUP.md. **Reversible:** per standing decision #1.

**D-55-D — Package renamed `app/mcp/` → `app/mcp_server/` to avoid shadowing the installed `mcp` SDK.** A local package named `mcp` made `from mcp.server.fastmcp import …` resolve to our package under pytest, breaking the import ("`mcp.server` is not a package"). **Reversible?** It's the canonical name now; trivial to rename again but no reason to.

**D-55-E — Two findings the planner didn't anticipate, both resolved:**
  - **Trailing slash:** mounting at `/mcp` with server `streamable_http_path="/"` means the live endpoint is `/mcp/` (a POST to `/mcp` 307-redirects). Clients must use the trailing slash. Recorded in CLIENT-SETUP.md.
  - **DNS-rebinding protection:** FastMCP validates the inbound `Host` header and 421s unknown hosts. For an internal, API-key-gated server reached over a trusted network this is friction (every client host would need enumerating), so I set `enable_dns_rebinding_protection=False` by default, overridable via `MCP_DNS_REBINDING_PROTECTION=true`. ⚠ **REVIEW:** if you want host-allowlisting on, flip that env and enumerate allowed hosts. Auth (the API key) is unaffected either way.

**D-55-F — `SKIP_MCP_LIFESPAN` test flag. ⚠ minor, REVIEW.** The MCP `StreamableHTTPSessionManager.run()` is single-use per instance, but the existing test suite creates one `TestClient(app)` per test, each entering the app lifespan. So conftest sets `SKIP_MCP_LIFESPAN=1` (read at import) to skip starting the MCP manager for plain REST tests (which never hit `/mcp`); the dedicated MCP test forces it back on for its single real-lifespan entry and restores it after. **Production never sets the flag** — it runs the MCP manager normally. **Why:** least-invasive way to keep 367 existing REST tests green without rewriting their fixtures. **Reversible?** Yes; an alternative is a session-scoped shared lifespan in conftest, more invasive.

**D-55-G — Pre-existing test flakes NOT caused by v8.0.** Running the full suite shows 4 `test_yolo_integration.py` failures under certain orderings — but they fail identically with my v8.0 tests REMOVED, and pass 7/7 in isolation. ### Phase 56 — Job registry + job_status + first AI tool

**D-56-A — Job registry is IN-MEMORY with TTL, not a DB table.** Jobs are short-lived (one generation, polled within the same agent session); an internal tool doesn't need them to survive a restart (the agent just re-fires). A DB table + migration would be over-engineering. Shape: module-level dict `job_id -> {status, result, error, created_at}` under an asyncio lock, with a TTL sweep. status ∈ pending/running/done/error. **Reversible?** Yes — swapping to a table later is localized; the `job_id`+`job_status` contract is stable. **Multi-worker caveat:** in-memory jobs are per-worker; the app runs a single uvicorn worker (internal tool), so fine — move to Redis/DB only if scaled.

**D-56-B — AI calls are already async → job runner uses asyncio.create_task, not to_thread for the AI.** `regenerate_single_scene`/`chat_completion` are already async. The runner: build context from a short-lived DB session and return job_id fast; an asyncio task runs the async generation WITHOUT holding a DB session across the await; store the result in the registry. The brief DB reads/writes aren't held across the AI await, so the event loop stays responsive under concurrent generations (criterion 4).

**D-56-C — First AI tool = `screenplay_generate_scene`**, wrapping the existing v6.0 `regenerate_single_scene` path. Returns `{job_id}` immediately; the regenerated-scene PREVIEW is retrieved via `job_status`. Proves the long-running pattern reused by Phases 58-60. (Persisting a kept version is handled with the Phase 58 screenwriting tools.)

### Phase 57 — Management tools

**D-57-A — `project_create` uses the `template` column, NOT the legacy `framework` enum. ⚠ FOUND A PRE-EXISTING BUG.**
- **Bug found:** the Postgres `framework` enum type stores lowercase labels (`three_act`), but SQLAlchemy's `Enum(Framework)` mapper uses uppercase member NAMES (`THREE_ACT`) for both write and read. So creating a Project with `framework=...` via the ORM fails on real Postgres (`invalid input value for enum framework: "THREE_ACT"`), and a raw lowercase write fails to read back. This pre-dates v8.0 — the legacy `framework` column is marked "kept for backward compat" and the live app uses the `template` column (which has `values_callable` and works). REST tests don't catch it because they run on sqlite (no PG enum enforcement).
- **Decision:** the MCP `project_create` tool creates projects the way the app actually does — sets `template=SHORT_MOVIE` (the working enum) and records the agent's requested framework in the JSON `template_config` column instead of the broken `framework` enum. The tool still accepts a `framework` arg (for the agent's intent) and reports it back. **I did NOT fix the global Framework column** — that's a schema migration touching other code paths, out of scope for an autonomous run; flagged here for you.
- **Reversible?** Yes. If you fix the Framework enum schema later, the tool can switch to setting it directly.
- ⚠ **REVIEW:** the legacy `framework` enum column is broken on real Postgres app-wide (not just MCP). Worth a small fix (add `values_callable` to the column + a data check) in a future cleanup.
