## Color

This phase operates in **breakdown mode** (`.breakdown-mode` CSS class), which uses a cool slate-blue palette. However, the staleness banner intentionally uses **amber** to signal warnings -- matching the existing `StalenessBar.tsx` pattern exactly.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | hsl(220 13% 5%) | Breakdown mode background |
| Secondary (30%) | hsl(220 12% 9%) | Cards, panels, center panel surface |
| Accent (10%) | hsl(213 80% 52%) | Breakdown mode interactive elements (buttons, links, toggles) |
| Warning | amber-500/10 bg, amber-500/20 border, amber-400 text | Staleness banner only |
| Destructive | hsl(0 72% 51%) | Not used in this phase |

Accent reserved for: mode toggle active state, primary action buttons, drag handle hover. Not used in the staleness banner.

Warning amber reserved for: staleness banner background, border, icon color, and text color. This matches the existing `StalenessBar.tsx` exactly:
- Background: `bg-amber-500/10`
- Border: `border-b border-amber-500/20`
- Icon + text: `text-amber-400`
- Dismiss button hover: `hover:text-amber-400` from `text-amber-400/60`

Source: StalenessBar.tsx (existing v2.0 component), index.css `.breakdown-mode` variables

## Interaction States

### Banner Lifecycle

| State | Condition | Visual |
|-------|-----------|--------|
| Hidden | `shotlist_stale === false` OR `shot_count === 0` | Banner not rendered |
| Visible | `shotlist_stale === true` AND `shot_count > 0` | Amber banner with warning icon + dismiss button |
| Dismissing | User clicked dismiss, API call in flight | Dismiss button shows disabled state (opacity-60) |
| Dismissed | API returns success, query invalidated | Banner disappears (re-render removes it) |
| Re-appears | User saves/generates script content while shots exist | Banner re-appears on next 30s poll or query invalidation |

### Data Flow

1. User edits script in Screenwriting mode and saves/generates
2. Backend hook sets `project.shotlist_stale = True`
3. User switches to Breakdown mode
4. React Query fetches `GET /api/shots/{project_id}/status` (or cache refreshes after 30s staleTime)
5. Banner renders above shotlist
6. User clicks X to dismiss
7. Frontend calls `POST /api/shots/{project_id}/acknowledge-stale`
8. Backend sets `project.shotlist_stale = False`
9. Query invalidation removes banner

## Accessibility

| Concern | Implementation |
|---------|---------------|
| Banner role | `role="alert"` on the banner container for screen reader announcement |
| Dismiss button | `aria-label="Dismiss staleness warning"` |
| Color contrast | amber-400 text on amber-500/10 background meets WCAG AA for large text; text-xs is small, so amber-400 on dark background (220 13% 5%) provides ~6.5:1 ratio (passes AA) |
| Keyboard | Dismiss button is a native `<button>`, focusable and activatable via Enter/Space |
| Motion | No animations -- safe for `prefers-reduced-motion` without additional handling |
