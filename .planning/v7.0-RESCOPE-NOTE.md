# v7.0 Breakdown Fidelity — Re-scope Note (autonomous decision)

**Date:** 2026-06-07
**Decided by:** Claude (autonomous v7.0 run, user asleep with "take the necessary decisions" authority)
**Why this exists:** A deep code exploration of the breakdown subsystem (before planning any phase) found the existing system is substantially more advanced than the v7.0 ROADMAP sketch (written 2026-06-06) assumed. Executing the original plan would mean building things that already exist. This note records the verified findings and the re-scope, so the user can review on waking.

## What I verified (against real source, not a summary)

1. **Extraction already uses full screenplay TEXT, not one-line summaries.**
   `breakdown_service.py:_build_extraction_context` sets `screenplay_texts=[sc.content for sc in screenplays]` and `_build_user_prompt` emits a `## Screenplay Content` section with the full text of every scene (plus a `## Scenes` summary list for indexing). → **BFID-01 as literally written ("extract against scene text not summaries") is ALREADY SATISFIED.**

2. **The data model already supports per-appearance context end-to-end — but the last step discards it.**
   - `ExtractedElement.scene_appearances: List[ExtractedSceneAppearance]` where `ExtractedSceneAppearance` has `scene_index` + `context`.
   - The AI system prompt already asks for per-appearance context.
   - `ElementSceneLink` table already has a `context` column.
   - BUT `_map_scene_indices_to_ids` (line 457) flattens `scene_appearances` to a bare `List[str]` of scene IDs, and `_reconcile_scene_links` writes `context=""` (line ~359). **The AI's context is computed, returned, then thrown away.** → APPR-01/02 are ~90% plumbed; the gap is threading `context` through one boundary + surfacing it in the UI.

3. **Cross-scene consolidation already exists.** `_deduplicate_elements` merges same-name same-category elements and unions their `scene_appearances` (lines 230-249). → **APPR-03 is ALREADY SATISFIED.**

4. **Staleness + user-edit preservation already exist.**
   - `breakdown_stale` flag set by `phase_data._mark_breakdown_stale` on write/scenes changes and by wizard apply (incl. v6.0 keep-scene-version); cleared after a successful extract.
   - `_upsert_elements` honors `user_modified=True` (skips overwrite) and `is_deleted=True` (no resurrection); `_reconcile_scene_links` deletes only `source="ai"` links and preserves `source="user"` links.
   → **REEX-01 is ALREADY SATISFIED.** REEX-02 is largely satisfied; the remaining gap is ensuring manual element edits reliably set `user_modified=True` and that the merge story is tested/explicit.

5. **Categories are genuinely limited.** `BreakdownCategory` enum = {character, location, prop, wardrobe, vehicle} (5), hardcoded in the enum (database.py), the system prompt (breakdown_service.py), and `BREAKDOWN_CATEGORIES` (frontend constants.ts). → **CATG-01/02/03 are real, unbuilt work.**
