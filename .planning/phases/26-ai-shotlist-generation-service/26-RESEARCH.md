# Phase 26: AI Shotlist Generation Service - Research

**Researched:** 2026-03-20
**Domain:** AI-powered shotlist generation with structured output, smart merge, and delta migration
**Confidence:** HIGH

## Summary

Phase 26 adds a backend service that reads screenplay content and scene data, calls the AI with structured output to generate a complete shotlist, and merges the result into the database while preserving user-edited shots. This follows the exact same architectural pattern as the existing `BreakdownService` (Phase 11) -- structured Pydantic models for AI output, context building from database, upsert with user_modified protection, and a single-transaction commit.

The Shot model already exists with JSONB `fields` column, `source` column ("user"/"ai"), and `scene_item_id` FK. Phase 26 must add two boolean columns (`user_modified`, `ai_generated`) via a delta migration, create a new `ShotlistGenerationService` class, define structured output Pydantic models for the AI response, and wire up a new endpoint (or extend the existing shots router). The update_shot endpoint must also be modified to set `user_modified = True` on any manual edit.

**Primary recommendation:** Model the service as a near-copy of `BreakdownService` -- same pattern of context builder, structured AI call via `chat_completion_structured`, upsert/merge logic, and run recording. Use the existing `chat_completion_structured` from `ai_provider.py` which already handles both OpenAI and Anthropic structured output.

### Pattern 1: Structured AI Output (Mirror BreakdownService)
**What:** Define Pydantic models for the AI's expected output, call `chat_completion_structured`, get a validated Python object back.
**When to use:** Whenever AI must return structured data that maps to database operations.
**Example:**
```python
# Source: existing pattern from backend/app/services/breakdown_service.py
from pydantic import BaseModel, Field
from typing import List

class GeneratedShot(BaseModel):
    """A single shot generated by AI for a scene."""
    scene_index: int = Field(description="1-based index of the scene this shot belongs to")
    shot_size: str = Field(description="e.g., Wide, Medium, Close-Up, Extreme Close-Up")
    camera_angle: str = Field(description="e.g., Eye Level, Low Angle, High Angle, Dutch")
    camera_movement: str = Field(description="e.g., Static, Pan, Tilt, Dolly, Tracking, Crane")
    description: str = Field(description="What the camera sees in this shot")
    action: str = Field(description="What happens during this shot")
    script_excerpt: str = Field(description="The specific passage from the script this shot covers")
    ordering_rationale: str = Field(description="Why this shot is in this position within the scene")

class ShotlistGenerationResponse(BaseModel):
    """Complete AI-generated shotlist."""
    shots: List[GeneratedShot]
```

### Pattern 2: Context Builder (Same as Breakdown)
**What:** Gather all relevant data from the database into a dataclass, then format it into an AI prompt.
**When to use:** When AI needs project context to generate output.
**Example:**

@dataclass

### Pattern 3: Smart Merge (Upsert with Protection)
**What:** On regeneration, delete old AI-generated unmodified shots, insert new AI shots, preserve user-modified shots.
**When to use:** AISG-06 -- regeneration must not destroy user edits.
**Merge algorithm:**
```python
def _merge_shots(self, db, project_id, generated_shots, ctx):
    # 1. Query existing shots for this project
    existing = db.query(Shot).filter(Shot.project_id == str(project_id)).all()

    # 2. Partition: user_modified=True stays, ai_generated=True & user_modified=False gets deleted
    for shot in existing:
        if shot.ai_generated and not shot.user_modified:
            db.delete(shot)

    # 3. Insert new AI-generated shots
    # Assign sort_order per scene, AFTER any preserved user shots
    for gen_shot in generated_shots:
        new_shot = Shot(
            project_id=str(project_id),
            scene_item_id=scene_id,  # mapped from scene_index
            fields={...},
            source="ai",
            ai_generated=True,
            user_modified=False,
            ...
        )
        db.add(new_shot)
```

### Anti-Patterns to Avoid
- **Streaming the generation response:** Unlike breakdown chat (which streams for UX), shotlist generation is a batch operation. Use `chat_completion_structured` (non-streaming), not `chat_completion_stream`. The frontend (Phase 27) will show a loading spinner.
- **Per-scene AI calls:** Do NOT call AI once per scene. Send ALL scenes in a single prompt and get all shots back. This is cheaper, faster, and gives the AI cross-scene context for better shot ordering.
- **Modifying user_modified shots during merge:** Never update a shot where `user_modified=True` during regeneration. The merge must skip these entirely.
- **Hard-coding field names in the AI response:** Use the JSONB `fields` column pattern. The AI returns named fields (shot_size, camera_angle, etc.) which get stored as a dict in `fields`.

## Common Pitfalls

### Pitfall 1: Sort Order Conflicts After Merge
**What goes wrong:** After deleting old AI shots and inserting new ones, sort_order values may collide with preserved user_modified shots.
**Why it happens:** User-modified shots retain their original sort_order; new AI shots get assigned sort_order that overlaps.
**How to avoid:** After inserting new AI shots, re-number ALL shots for each scene: user_modified shots keep their relative position, AI shots fill gaps.
**Warning signs:** Shots appear in wrong order after regeneration.

### Pitfall 2: Scene Index Off-by-One
**What goes wrong:** AI returns 1-based scene indices but code uses 0-based list indexing.
**Why it happens:** Different conventions between AI prompt and Python.
**How to avoid:** Follow the BreakdownService pattern exactly: prompt says "Scene 1, Scene 2, ..."; AI returns 1-based; code does `zero_based = scene_index - 1`. Add bounds checking.
**Warning signs:** Shots assigned to wrong scenes.

### Pitfall 3: Large Script Token Overflow
**What goes wrong:** Long screenplays exceed the AI model's context window when combined with system prompt.
**Why it happens:** Full screenplay text + scene list + system prompt + response tokens > context limit.
**How to avoid:** Truncate screenplay text at a reasonable limit (same pattern as `OpenAIService.review_section` which truncates at `MAX_SECTION_LENGTH`). For shotlist generation, use a higher limit (e.g., 50,000 chars) since this is the core input. Set `max_tokens` appropriately for the response (shotlists can be large -- 8000+ tokens).
**Warning signs:** AI errors, truncated responses, missing scenes in output.

### Pitfall 4: Forgetting to Set user_modified on Manual Edit
**What goes wrong:** User manually edits a shot, but the flag stays False. Next regeneration deletes it.
**Why it happens:** The `update_shot` endpoint in `shots.py` doesn't set the flag because it doesn't exist yet.
**How to avoid:** Modify `update_shot` to always set `shot.user_modified = True` when any field is changed.
**Warning signs:** User-edited shots disappear after regeneration.

### Pitfall 5: SQLite Test Compatibility
**What goes wrong:** Tests fail because SQLite doesn't support `BOOLEAN` the same way as PostgreSQL.
**Why it happens:** SQLite stores booleans as integers (0/1), and the conftest.py patches UUID and Enum columns but doesn't need special handling for Boolean.
**How to avoid:** SQLAlchemy's Boolean type already handles SQLite. Just ensure the model column uses `Column(Boolean, default=False)`. No special patching needed in conftest.py.
**Warning signs:** Test failures on boolean comparisons.

## Open Questions

1. **Shot Number Assignment Strategy**
   - What we know: Existing shots use `shot_number` field. User-modified shots have their own numbers.
   - What's unclear: After merge, should AI renumber all shots sequentially, or preserve user shot numbers and fill gaps?
   - Recommendation: Renumber within each scene group after merge (1, 2, 3...) since shot_number is a display label, not an ID. Use sort_order for actual ordering.

2. **Maximum Shots Per Scene**
   - What we know: No current limit on shots per scene.
   - What's unclear: Should the AI be instructed to generate a specific range of shots per scene?
   - Recommendation: Let AI decide naturally based on scene complexity. Typical range is 3-10 shots per scene. Add a soft cap in the prompt ("typically 3-10 shots per scene, more for complex action sequences").

3. **Response Token Budget**
   - What we know: Each shot is ~100-150 tokens in structured output. A 10-scene screenplay with 5 shots/scene = 50 shots = ~5,000-7,500 tokens.
   - What's unclear: Will very long screenplays overflow the response token limit?
   - Recommendation: Use `max_tokens=8000` for the AI call. For screenplays with >15 scenes, consider warning the user or batching (but start without batching -- YAGNI).
