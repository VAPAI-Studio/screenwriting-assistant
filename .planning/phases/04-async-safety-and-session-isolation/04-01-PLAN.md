<objective>
Fix the shared DB session bug (REVW-05) across all three `asyncio.gather` sites in `agent_service.py` by refactoring to a session-per-task pattern using a `session_factory` callable. This eliminates intermittent `DetachedInstanceError` and `MissingGreenlet` exceptions when multiple agents run concurrently.

Purpose: Phase 5 (Agent Review Middleware) depends on `run_multi_agent_review` working safely under concurrency. The same bug also affects the orchestrator chat flow (`_orchestrate` and `_orchestrate_stream_prepare`), which is a production issue today.
Output: Three refactored gather sites, updated callers, and a test suite proving session isolation.
</objective>

```python
async def _review_with_session(self, agent: Agent, section: Section, project: Project, session_factory: SessionFactory) -> Dict:
    """Per-task session wrapper for review_section."""
    db = session_factory()
    try:
        return await self.review_section(agent, section, project, db)
    finally:
        db.close()

async def _get_specialist_context_with_session(self, agent: Agent, query_text: str, owner_id, session_factory: SessionFactory) -> Dict:
    """Per-task session wrapper for _get_specialist_context."""
    db = session_factory()
    try:
        return await self._get_specialist_context(agent, query_text, owner_id, db)
    finally:
        db.close()
```

**IMPORTANT anti-patterns to avoid:**
- Do NOT set `session_factory=SessionLocal` as a default parameter in method signatures -- import and pass explicitly at call sites for testability (per research)
- Do NOT reload ORM objects inside per-task sessions -- `agent`, `section`, `project` have already-loaded attributes that are safe to read. Only use the per-task `db` for fresh queries (which `review_section` and `_get_specialist_context` do via `rag_service`)
- Use `try/finally` (not `try/except` or context manager) for session cleanup -- `finally` runs even on `asyncio.CancelledError`
