<objective>
Make scene-by-scene screenplay generation continuity-aware. Thread a running prose synopsis ("story so far") plus the full text of the immediately preceding scene through the existing sequential loop in `_generate_scripts`, inject both into each scene prompt (conditionally — first/single scene gets none), regenerate the synopsis after each successful scene via a separate `chat_completion` call, and persist the final synopsis into the existing `screenplay_editor` PhaseData JSON.

Purpose: Tone, voice, and setup/payoff stay consistent across the scene sequence because each scene is written with awareness of what was actually written before — not just one-line plan summaries (CONT-01, CONT-02, CONT-03).

Output: Modified `_generate_scripts` with continuity threading and a `_update_synopsis` helper; modified `script_writer_wizard` persistence; a new test module covering the four success criteria.
</objective>

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-45-01 | Tampering / Injection | prior generated scene text + synopsis flowing into later scene prompts | accept | Prior-scene text and synopsis are model-generated within the same trusted internal flow; no untrusted user input enters here that did not already enter the existing scene prompt. No new injection surface vs. baseline. |
| T-45-02 | Denial of Service | token/cost blowup from unbounded synopsis or verbatim prev-scene text accumulating across many scenes | mitigate | Synopsis is re-summarized to a fixed word cap each scene (D-03) — it does not grow unbounded. Only the SINGLE immediately-preceding scene's text is injected verbatim (D-01), not all prior scenes, so per-prompt size is bounded. Synopsis-update call uses a small max_tokens. |
| T-45-03 | Tampering | failed-scene error placeholder poisoning continuity context | mitigate | prev_scene_text and synopsis advance only inside the success branch; a failed scene's `[Generation failed: ...]` placeholder is never propagated into the next prompt or the synopsis (Task 1 + Task 3 test 4). |
| T-45-04 | Information Disclosure | synopsis persisted into JSONB | accept | Synopsis is derived solely from content the user already owns and which is already persisted (screenplays in the same content dict). No new data class or PII introduced. |
| T-45-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed — chat_completion is consumed as-is; no new dependency. Package Legitimacy Gate not applicable (no install tasks). |

No HIGH-severity threats: the surface is internal prompt construction plus a bounded JSONB write, with no new external/auth/dependency surface and explicit bounds on token growth and failed-scene propagation.
