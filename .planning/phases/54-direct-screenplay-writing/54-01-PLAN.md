<objective>
Make the Screenplay Editor writable from an empty project and persist hand-written screenplays exactly like generated ones.

Today the editor hard-blocks when there is no content ("Generate screenplays using the Script Writer Wizard first") and the PATCH save endpoint 404s when the `screenplay_editor` PhaseData does not yet exist — so a from-scratch screenplay can neither be entered nor saved, and even if saved would not feed the breakdown (which reads `ScreenplayContent` rows the generic PATCH never creates).

This plan: (1) makes the PATCH handler a true upsert (fetch-or-create, copying the proven wizard pattern); (2) adds a screenplay-scoped branch in PATCH that idempotently reconciles `ScreenplayContent` rows from the saved screenplays and marks breakdown/shotlist stale; (3) replaces the dead-end empty state with a "Start writing" affordance; (4) adds a heading-based splitter for the zero-originals case so hand-written text splits into INT./EXT. scenes (or one "Untitled" scene).

Purpose: deliver WRITE-01..04 — direct screenplay authoring that round-trips and flows into breakdown.
Output: upserting/reconciling PATCH handler, writable editor empty state + pure heading splitter, and backend tests proving no-404 / sync / idempotence / staleness / generic-non-sync.
</objective>

<design_decision>
**ScreenplayContent sync location (D-54-05): chosen option (b) — guarded branch inside the generic PATCH handler.**

Rationale: fewer moving parts than a dedicated endpoint, and it requires NO frontend change — the existing `api.updateSubsectionData` PATCH client already targets `PATCH /phase-data/{project}/write/screenplay_editor`. The generic upsert (fetch-or-create) is safe for ALL subsections; only the `ScreenplayContent` reconcile + staleness is gated behind `if phase == "write" and subsection_key == "screenplay_editor"`. The generic PATCH stays generic for every other subsection (it must NOT create ScreenplayContent for arbitrary subsections — that is the design constraint and is covered by a test).

**Reconcile policy (D-54-05): delete-then-recreate this project's ScreenplayContent rows in one transaction.** The wizard apply path APPENDS (a documented duplicate-row problem); the manual-save path REPLACES so repeated saves stay idempotent. Delete `ScreenplayContent` where `project_id == this project`, then recreate one row per saved screenplay with `content=sp["content"]` and `formatted_content=sp` (sp carries `episode_index`), all before the single `db.commit()`.
</design_decision>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → PATCH /phase-data | Authenticated, owner-scoped write of screenplay JSONB; untrusted free-text body crosses here |
| PATCH handler → ScreenplayContent rows | Server-side write the breakdown extraction later reads |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-54-01 | Tampering | ScreenplayContent reconcile on save | mitigate | Delete-then-recreate scoped to `project_id` of the verified-owned project (D-54-05); cannot touch other projects' rows. Idempotent so repeated saves cannot accumulate duplicate rows (the documented v6.0 duplicate-row problem). Covered by the idempotence test. |
| T-54-02 | Elevation of Privilege | PATCH upsert creating PhaseData for arbitrary subsection | accept | Upsert is gated by `_verify_project_ownership` (unchanged); a user can only create subsections under projects they own. | T-54-03 | Tampering | Generic PATCH staying generic | mitigate | ScreenplayContent sync gated behind `if phase == "write" and subsection_key == "screenplay_editor"`; a test asserts a non-screenplay subsection PATCH creates zero ScreenplayContent rows, so the generic path cannot be coerced into writing screenplay rows. |
| T-54-04 | Denial of Service | Large hand-written screenplay body | accept | Existing RequestSizeLimitMiddleware (10MB) and per-IP rate limiting (60 req/min) already bound request size and frequency; no new surface. |

No HIGH-severity threats. No package installs in this phase (no Package Legitimacy Gate required).
