# Phase 49 Plan 01: Backend Single-Scene Regenerate + Keep Summary

Backend slice of EVAL-01: a single scene can be regenerated by `episode_index` through the improved phases-45-48 path, previewed without clobbering the stored version, and explicitly persisted — via a shared `_generate_one_scene` helper and two owner-scoped endpoints, with all 27 prior tests staying green.

## What Was Built

**Task 1 — `_generate_one_scene` helper + `regenerate_single_scene` (commit 34b8642):**
Extracted the per-scene loop interior of `_generate_scripts` (prompt construction, the `chat_completion` call at `temperature=0.7, max_tokens=4000, json_mode=False`, the fence-tolerant native parse, the `TITLE:` split, and the empty-title→summary fallback) **verbatim** into a private `async _generate_one_scene(...)` returning `{title, content, episode_index}` (or the `[Generation failed: ...]` failure-branch dict on exception). `_generate_scripts` now delegates per-iteration and retains the success-only continuity advance (`_update_synopsis` only when the returned dict has no `error` key), the `if not episodes` guard, and the exact `{screenplays, synopsis}` return. Added public `regenerate_single_scene(config, project_context, episode_index, synopsis, prev_scene_text)` that builds `character_section`/`scene_outline` identically, bounds-checks the index (raises `ValueError`), and calls the same helper — and does NOT advance/rewrite the synopsis (D-49-05).

**Task 2 — preview + keep endpoints + schemas (commit 4fc7f8d):**
- `POST /api/wizards/regenerate-scene` — owner-scoped; builds the regenerate config from the `scenes/scene_list` ListItems (ordered by `sort_order`) + character data + carried `runtime_target`/`custom_guidance` (best-effort from the latest `script_writer_wizard` WizardRun); sources `synopsis` and `prev_scene_text` from the stored `screenplay_editor` content; returns the regenerated scene as a preview and writes NOTHING (D-49-02).
- `POST /api/wizards/keep-scene-version` — owner-scoped; replaces `screenplays[episode_index]` in `screenplay_editor` PhaseData.content (`flag_modified`), updates the matching `ScreenplayContent` row (match by `formatted_content.episode_index`, fallback to stable ordering — no migration, D-49-03), marks breakdown + shotlist stale, leaves `content["synopsis"]` untouched (D-49-05), returns `{status, episode_index}`.

## Anchor / Contract Preservation (REFACTOR SAFETY)

Moved byte-for-byte into `_generate_one_scene`: the SHORT user-prompt opener "You are an expert screenwriter."; the LONG SYSTEM string "...native plain text only..."; the SCENE_MARKER "YOUR TASK: Write scene"; `json_mode=False`; the TITLE parser + fence tolerance + empty-title→summary fallback; the `{title,content,episode_index}` per-item shape. `_generate_scripts` preserves the `{screenplays, synopsis}` return, the success-only `_update_synopsis` advance, the `[Generation failed:]` branch, and the `if not episodes` guard. The two distinct prompt strings (short user opener vs. long system message) were moved unchanged and not conflated (PLAN-CHECKER WARNING 1).

**Alignment assumption (documented in code + here, PLAN-CHECKER WARNING 2):** `episode_index` indexes BOTH the `sort_order`-ordered `scene_list` (regenerate `episodes`) AND `screenplays[]` 1:1 — the same implicit contract breakdown/shotlist already rely on. Both endpoints bounds-check `episode_index` and return a clean 404 when out of range.

### Auto-fixed Issues

**1. [Rule 1 - Bug] Repointed the craft source-pin test after the refactor**
- **Found during:** Task 1 GREEN
- **Issue:** `test_craft_guidance.py::test_craft_block_present_in_generate_scripts_source` uses `inspect.getsource(template_ai_service._generate_scripts)` to pin the craft anchors. - **Files modified:** `backend/app/tests/test_craft_guidance.py`
- **Commit:** 34b8642

**2. [Rule 3 - Blocking] Owner filter uses `str()` coercion instead of raw-UUID compare**
- **Found during:** Task 2 GREEN (endpoint tests returned 404 "Project not found")
- **Issue:** The plan said to copy `run_wizard`'s owner filter verbatim (`id == request.project_id, owner_id == current_user.id` with raw UUID objects). Under the SQLite test harness the UUID columns are patched to `String(36)`; a raw-UUID-object WHERE bind does not round-trip to the stored string, so the owner lookup never matched (404). `run_wizard` itself is not exercised via the client under SQLite, so this latent harness incompatibility was never observed there.
- **Fix:** Used `id == str(request.project_id)` and `owner_id == str(current_user.id)` — the established codebase convention (`phase_data._verify_project_ownership`, `breakdown.py`). This is Postgres-safe (native UUID columns accept string equality) AND SQLite-test-safe. Semantics are identical to the verbatim filter; only the bind type changed.
- **Files modified:** `backend/app/api/endpoints/wizards.py`
- **Commit:** 4fc7f8d
