<objective>
Make run_migrations() self-sufficient on a fresh managed Postgres (Railway): when the `projects` table is absent, run the idempotent init_db.sql baseline as step 0 before applying deltas; guard the whole run with a Postgres advisory lock so concurrent instances serialize DDL; fail hard on any baseline or delta failure. Add focused tests (DMIG-01).

Purpose: On Docker, the entrypoint hook already creates the schema, so step 0 is skipped (no double-run). On Railway there is no such hook, so step 0 fires once on first boot — the prerequisite that lets later phases target a real host with no manual migration step.
Output: Extended db_migrator.py + new test_db_migrator.py.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| repo SQL files → live Postgres on boot | init_db.sql and delta/*.sql are executed as DDL at startup; their content is trusted iff the repo is trusted |
| concurrent app replicas → shared DB | multiple instances may boot simultaneously (Railway scaling/redeploy overlap) and contend on DDL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-62-04 | Tampering | init_db.sql / delta/*.sql executed on boot | accept | Repo-trust boundary: SQL is version-controlled and reviewed; no untrusted input is interpolated into the SQL (file contents executed verbatim). CI/code-review is the control; no runtime user input reaches these files. |
| T-62-05 | Denial of Service / Integrity | concurrent replicas racing on DDL | mitigate | Wrap the whole migration run in pg_advisory_lock(<fixed key>) acquired first and released in finally (D-05); other instances block then find nothing pending — prevents partial/duplicate DDL. |
| T-62-06 | Integrity | partial migration leaving a half-built schema | mitigate | Fail hard on baseline or delta error (D-04, D-06) — boot crashes loudly in Railway logs rather than serving against a half-migrated DB; per-migration commit means the failing migration retries cleanly next boot. |
| T-62-SC | Tampering | package-manager installs | n/a | No new packages installed in this phase (uses existing sqlalchemy + stdlib). No legitimacy gate required. |
</threat_model>
