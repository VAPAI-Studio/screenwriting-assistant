### Integration Gaps (2 found)

**1. Migration 008 not in docker-compose.yml** (Medium severity)

`docker-compose.yml` mounts migrations 001–007 but not `008_agent_pipeline_maps.sql`. On a fresh `docker compose up --build`, the SQL migration path won't create the `agent_pipeline_maps` table.

**Mitigation:** `init_db.py` calls `Base.metadata.create_all()` which creates the table from the ORM model. So the table IS created at app startup — just not via the migration file. This is an inconsistency between SQL-file and ORM init paths, not a hard blocker.

**Fix:** Add one line to docker-compose.yml volumes:
```
- ./backend/migrations/008_agent_pipeline_maps.sql:/docker-entrypoint-initdb.d/008_agent_pipeline_maps.sql
```

**2. Frontend WizardRunResponse missing agents_consulted** (Low severity)

Backend `WizardRunResponse` Pydantic schema includes `agents_consulted: List[Dict]` populated via `model_validator`. Frontend TypeScript `WizardRunResponse` interface in `template.ts` does not declare this field. The data is returned by the API but invisible to the frontend type system.

**Impact:** No frontend component renders which agents reviewed the wizard output. Display-only gap — backend wiring and data are correct.
